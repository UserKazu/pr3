# Звіт до Практичної роботи №3
Тема: Розгортання Application Server на базі NodeJS та Fastify
Виконав: Пловайко Вадим, група КН-41

## Мета роботи
Отримати практичні навички інтеграції Fastify у існуючий стек із Docker Compose, налаштувати проксі через Nginx та реалізувати базовий набір RESTful-ендпоінтів (CRUD).

## Короткий опис виконаної роботи
1. Повторив основи Node.js і асинхронної моделі подій (event loop).
2. Ознайомився з Fastify — обрав цей фреймворк за його продуктивність і зручну систему схем для валідації.
3. Замінено (умовно) попередній сервер на Fastify; налаштовано маршрути та схеми валідації для кожного ендпоінту.
4. Оновлено Dockerfile та docker-compose.yml для запуску сервісу в контейнері й проксі через Nginx.
5. Реалізовано п'ять основних ендпоінтів: список, отримання по id, створення, оновлення (PUT/PATCH) та видалення.
6. Проведено тестування через Nginx (локально на порті 80) — перевірено статус-коди та випадки коли ресурс не знайдений.

## Зміни в конфігурації
- `Dockerfile`: образ node:24, робоча директорія `/usr/src/app`, expose 3000.
- `docker-compose.yml`: сервіси `app` і `nginx`, спільна мережа `webnet`, nginx проксіює `/api/` на `app:3000`.
- `nginx/default.conf`: правило location /api/ з proxy_pass на внутрішній хост.

## Опис реалізації ендпоінтів
- GET /api/resources — повертає масив ресурсів.
- GET /api/resources/:id — повертає один ресурс або 404.
- POST /api/resources — створює ресурс з required-полями `name` і `type`.
- PUT /api/resources/:id — повна заміна ресурсу.
- PATCH /api/resources/:id — часткове оновлення (price, amount).
- DELETE /api/resources/:id — видалення ресурсу.

## Тестування (коротко)
- Перевірено запити через Nginx: `GET http://localhost/api/resources` — 200 OK.
- Створення: `POST /api/resources` з валідним тільом — 201 Created.
- Оновлення та видалення перевірені локально та через проксі.

## Висновки
Fastify показав себе як легкий і продуктивний фреймворк з корисною підтримкою схем валідації. Nginx як реверс-проксі добре доречний для приховування внутрішніх портів контейнерів і обробки статичних ресурсів. Для подальшого розвитку проєкту рекомендую підключити СУБД (наприклад, PostgreSQL або MongoDB) замість файлового сховища.
